# FFI ABI 收敛改造计划

## 任务目标

本次任务目标是围绕当前 `vulcan-luaskills` 的既有 FFI 接口，直接对现有 ABI 做收敛改造，而不是并行设计一套独立 `v2`。改造必须满足以下边界：

1. 不影响当前 Rust 直连集成方式与主运行时行为。
2. 以 `beta / v0.1.x` 阶段可接受的破坏性整理为前提，直接把现有 FFI 往正式产品接口方向收敛。
3. 优先解决 FFI 的所有权模型、错误模型、callback 契约、线程与重入约束表达不清的问题。
4. 文档、头文件、实现、测试必须同步推进，避免“代码变了但对接规则没变成正式契约”。

## 详细执行步骤

1. 盘点现有 FFI 接口的 ABI 形态，确认哪些入口仍然依赖 `char * / char ** / error_out` 模式，哪些入口已经可以在不影响 Rust 主集成的前提下调整。
2. 将已有 `FFI_V2_DRAFT` 思路转化为“现有 FFI 收敛改造方案”，不再按并行版本语义推进，而是提炼出当前版本可直接实施的改造项。
3. 第一阶段先统一低风险但收益高的部分：
   - 明确 unsafe 边界
   - 固化所有权与释放规则
   - 固化 callback 注册顺序、异常边界、同线程重入限制
4. 第二阶段开始 ABI 结构收敛：
   - 评估并设计统一的 buffer / status 结构
   - 优先改造最容易误用的 callback 返回模型
   - 对现有 `char **error_out` 与多 out 参数模式给出替代方案
5. 第三阶段补齐外部交付物：
   - 重写 FFI 对接文档
   - 更新 `include/vulcan_luaskills_ffi.h`
   - 明确 beta 发布边界、宿主责任边界、错误使用边界
6. 每完成一批实现后，执行 `cargo check`、`cargo test` 验证 Rust 主集成未受影响。

## 技术选型与改造原则

### 1. 总体原则

- 直接收敛现有 FFI，不维护并行 `v1/v2` 双版本体系。
- 保持 Rust crate 主路径稳定，把变更控制在 FFI 导出面、头文件和跨语言桥接层。
- 对外 ABI 优先显式表达所有权和长度，不继续扩大依赖隐式 NUL 终止字符串的范围。

### 2. ABI 收敛方向

- 以统一 buffer 结构替代部分 `char *` 与 `uint8_t *` 离散返回。
- 以统一 status / error 结构替代大量 `error_out` 分支模式。
- 优先改造 callback 相关 ABI，因为这里是误用成本最高的部分。
- 对已经没有第三方接入的接口，允许在 `beta / v0.1.x` 阶段直接做破坏性调整。

### 3. 文档原则

- 文档不再按“未来 v2”口径描述，而是按“当前正式 FFI 契约正在收敛”来组织。
- 所有宿主必须遵守的规则都写成硬约束，不再写成建议。
- 明确哪些问题属于库内部保证，哪些问题属于宿主初始化顺序、callback 实现方式或 FFI 误用。

## 验收标准

1. 形成一份清晰的现有 FFI 收敛改造方案文档，可直接作为后续社区协作基线。
2. 至少完成一批实际 ABI 重构，不停留在纯文档或纯讨论层。
3. FFI 文档、头文件、实现保持一致，不出现文档与代码契约不一致的情况。
4. `cargo check` 与 `cargo test` 通过，确认当前 Rust 主集成未被破坏。
5. 任务完成后输出结构化执行变更总结，并将计划归档到 `docs/completed/20260422/`。
